# 项目展示与面试讲解

## 1. 简历项目名称

**带认证、库存扣减、订单状态机和幂等控制的 Go 订单库存管理系统**

技术栈：Go、Gin、GORM、MySQL、Redis、RabbitMQ、JWT、Docker、React、TypeScript

## 2. 简历项目描述

基于 Go、Gin、GORM、MySQL、Redis 和 RabbitMQ 实现轻量级订单库存管理系统，覆盖 JWT 认证、最小 RBAC、商品与库存管理、幂等下单、订单状态流转、超时自动取消、库存流水及 React 管理台。项目重点处理订单与库存多表事务一致性、并发防超卖、重复请求和用户订单数据隔离，并通过集成测试与 CI 验证关键业务链路。

## 3. 简历职责与亮点

- 按 handler、service、dao、model 分层实现商品、库存、订单和用户模块，统一参数校验、业务错误码及 API 响应结构。
- 设计订单创建事务，将幂等记录、订单主表、订单明细、库存扣减和库存流水纳入同一事务，异常时整体回滚。
- 使用库存行锁配合 `stock_quantity >= quantity` 条件更新防止超卖，并通过并发下单和多商品回滚测试验证最终库存一致性。
- 基于用户级复合唯一索引和 SHA-256 请求摘要实现创建订单幂等：相同请求返回原订单，不同请求复用 Key 返回冲突。
- 使用条件更新实现订单状态机，限制待支付、已支付、已完成和已取消之间的状态流转；取消订单时在事务中回补库存并写流水。
- 实现 JWT 登录认证和订单归属隔离；跨用户访问统一按资源不存在处理，降低订单信息泄露风险。
- 使用数据库实时角色查询实现最小 RBAC，限制商品、库存和库存流水管理接口仅管理员访问。
- 使用事务 Outbox、RabbitMQ TTL/DLX、Publisher Confirm 和手动 ACK 实现待支付订单超时取消，并通过条件更新保证重复消息不重复回补库存。
- 使用 Redis cache-aside 缓存商品详情，在商品状态变更时主动失效；Redis 异常时降级查询 MySQL。
- 使用 Goose、Docker Compose 和 GitHub Actions 完成数据库迁移、依赖编排以及 lint、test、race、vet、build 检查。

> 简历只保留 4～5 条与目标岗位最相关的内容。没有压测数据时不要写“性能提升百分比”或“支撑某个 QPS”。

## 4. 一分钟项目讲解

这是一个订单库存管理系统，主要不是展示 CRUD，而是处理下单链路中的一致性问题。用户通过 JWT 登录后只能访问自己的订单，商品和库存写接口再通过数据库实时角色查询限制为管理员。创建订单时，服务端先用用户 ID 和幂等 Key 的复合唯一索引抢占处理权，再对规范化订单项计算请求摘要。幂等记录、订单、订单项、库存扣减、库存流水和超时 Outbox 在同一个 MySQL 事务中提交；库存读取使用行锁，实际扣减还带库存下限条件。待支付订单通过 RabbitMQ TTL/DLX 延迟投递触发取消，消费者用状态条件更新保证重复消息不会重复回补库存。项目还提供 Redis 商品缓存、React 管理台、数据库迁移、Docker 编排和 CI。

## 5. 核心设计怎么讲

### 5.1 为什么库存扣减同时使用行锁和条件更新

问题：并发请求先查库存再更新，可能都读到相同库存并发生超卖。

原因：普通读取和后续写入之间存在竞争窗口，多条商品明细还需要获得准确的扣减前后值以记录流水。

修改建议：事务内使用 `SELECT ... FOR UPDATE` 串行化同一库存行的计算，再用 `stock_quantity >= quantity` 的条件更新作为最终保护；通过受影响行数判断库存是否充足。

示例：库存为 1 时两个请求同时购买 1 件，只有一个事务能成功扣减，另一个请求返回库存不足，最终库存不会小于 0。

### 5.2 为什么不能只用 idempotency_key 唯一索引

问题：同一个 Key 可能被客户端错误地用于不同订单内容。

原因：唯一索引只能判断 Key 是否重复，不能判断两次请求的业务语义是否相同；全局唯一 Key 还会让不同用户互相冲突。

修改建议：唯一范围使用 `(user_id, idempotency_key)`，并对排序后的订单项计算 SHA-256 摘要。Key 与摘要都相同才返回原订单，Key 相同但摘要不同则返回 HTTP 409。

示例：用户第一次用 Key `create-001` 购买商品 A，重试同一请求返回原订单；若改为商品 B 仍使用该 Key，则判定为幂等冲突。

### 5.3 为什么状态更新要带旧状态条件

问题：先查询状态、再无条件更新，会在支付和取消并发时互相覆盖。

原因：查询结果在更新前可能已经失效，仅在 service 层判断不能保证数据库写入时状态仍然合法。

修改建议：执行 `UPDATE orders SET status = ? WHERE id = ? AND user_id = ? AND status = ?`，并检查受影响行数。这样状态校验和写入在数据库中原子完成。

示例：支付和取消同时竞争待支付订单时，只有一个条件更新成功，订单不会同时出现已支付状态和取消库存回滚结果。

### 5.4 Redis 为什么不参与下单库存

问题：如果缓存与数据库共同维护可售库存，会引入双写一致性和故障恢复复杂度。

原因：当前项目规模下，MySQL 行锁和条件更新已经能正确保证库存；过早引入 Redis 预扣会增加补偿、对账和消息可靠性问题。

修改建议：Redis 只承担商品详情 cache-aside 缓存，库存正确性以 MySQL 为准。只有明确出现数据库库存热点后，再评估 Redis 预扣、消息队列和异步落库方案。

示例：Redis 不可用时商品详情查询降级到 MySQL，下单事务不受缓存状态影响。

## 6. 高频追问

### 创建订单事务失败后，幂等 Key 为什么还能重试？

幂等记录和订单写入处于同一事务。任何订单项、库存扣减或流水写入失败都会让事务整体回滚，首次插入的幂等记录也随之回滚，因此客户端可以使用原 Key 重试。

### 多商品订单如何避免死锁？

当前请求摘要会排序，但实际库存加锁仍按客户端订单项顺序执行。不同订单若以相反顺序购买多个相同商品，理论上可能发生死锁；MySQL 会回滚其中一个事务。更稳妥的演进方式是在进入事务前按 `product_id` 排序并统一加锁顺序，同时对可重试死锁做有限次数重试。

### 取消订单为什么放在事务里？

订单状态、多个商品库存和多条库存流水必须保持一致。如果任一库存回补或流水写入失败，状态变更也应回滚，避免订单显示已取消但库存没有完全恢复。

### JWT 方案还有什么边界？

当前实现支持签发、签名算法限制、issuer 和过期时间校验，但没有刷新令牌、主动注销黑名单和密钥轮换。项目实现的是最小 RBAC：管理接口实时查询数据库角色，但没有角色管理 API、多角色合并规则或权限点模型。

### 项目目前最值得继续补什么？

优先补可观测性、管理员操作审计和部署运行证据。若面向更高并发场景，再统一多商品加锁顺序、增加死锁重试，并基于压测结果决定是否引入 Redis 库存预扣。

## 7. 演示顺序

1. 注册并登录，展示 Bearer Token 和受保护接口。
2. 创建商品、初始化库存并上架。
3. 使用固定幂等 Key 创建订单，观察库存扣减和 `biz_type=3` 流水。
4. 重放相同请求，展示返回同一订单且库存不重复扣减。
5. 用相同 Key 修改订单项，展示幂等冲突。
6. 取消待支付订单，展示状态变化、库存回补和 `biz_type=4` 流水。
7. 演示待支付订单不能直接完成、已支付订单不能取消。
8. 切换用户访问原订单，展示订单归属隔离。
9. 使用普通用户和管理员分别调用商品写接口，展示最小 RBAC。
10. 使用短超时测试配置演示待支付订单自动取消和库存回补。

完整请求可使用 [http/demo_flow.http](http/demo_flow.http)，测试证据和限制以 [test_result.md](test_result.md) 为准。

## 8. 表述边界

- 可以说“通过并发集成测试验证防超卖”，不要说“已支撑生产级高并发”。
- 可以说“实现数据库实时校验的最小 RBAC”，不要说“完整权限平台”，因为当前没有角色管理 API 和细粒度权限点。
- 可以说“Redis 缓存异常可降级”，不要说“分布式缓存高可用”，因为当前没有 Redis Cluster 或 Sentinel。
- 可以说“创建订单支持请求幂等”，不要泛化为“所有写接口都支持幂等”。
- 可以说“CI 包含 race 检查”，不要说“项目不存在数据竞争”；race detector 只能覆盖实际执行路径。
